# 性能优化工作流

## 适用范围
- 该工作流为通用流程，适用于各类性能敏感系统

## 项目约定（必选）
- 所有测试用例必须放在项目根目录 `test/` 下。
- 每次优化完成后，必须在 `perf_report/` 下生成一份 markdown 报告。
- 所有性能/压测输出必须存放在 `perf_report/baseline/` 下。

## 语言规范（必选）
- 在整个工作流中，默认使用中文沟通与输出。
- `todo.md`、`perf_report/*.md`、优化结论、风险说明、回滚说明、测试结论必须使用中文。
- 仅以下内容可保留英文：代码标识符、命令行、文件路径、协议字段名、第三方工具原生输出。
- 若用户明确要求其他语言，按用户要求执行。

## 1）先确认扫描范围，再分析并产出 TODO（第一步，必选）
- 在扫描优化项之前，先让用户选择扫描范围：
  - `entire-project`：扫描整个仓库
  - `module-scope(<path-or-module>)`：只扫描指定模块或路径
- 若范围不明确，在用户确认前不要做大范围分析。
- 基于选定范围，读取项目结构、热点模块、基准/测试文件和现有文档。
- 识别候选优化点，并评估收益与风险。
- 生成或更新 `todo.md`，按优先级分组（例如 P0/P1/P2）。
- 每个 TODO 项必须包含唯一且稳定的 ID（例如 `OPT-P0-001`、`OPT-P1-002`），并且该 ID 是唯一可接受的执行选择器。
- 每个项应包含：
  - 预期收益（throughput/latency/cpu/memory/stability）
  - 可能触达点（文件/模块）
  - 验证方法（benchmark/test/metric）
- 要求用户按 ID 选择要执行的 TODO 项。
- 在用户输入一个或多个 TODO ID 前，不得开始代码优化。
- 若用户要求自动优先级排序，先给出推荐 ID，再等待用户输入 ID 后开始执行。

## 2）为所选 TODO ID 定义目标与约束
- 明确目标指标（吞吐、延迟、CPU、内存、尾延迟、稳定性）。
- 明确约束条件（正确性、一致性、资源限制、兼容性）。
- 编码前先设定成功阈值。

## 3）压测前检查测试/基准覆盖（必选）
- 在任何压力或性能测试前，先扫描 `test/` 下已有测试/基准，确认覆盖目标优化路径。
- 测试工具策略：
  - 压力/性能测试必须使用 **Google Benchmark**（benchmark target + filter）。
  - 功能/单元验证必须使用 **GTest**（gtest target/test case）。
- 若没有合适 benchmark，先在 `test/` 下补一个最小 Google Benchmark 用例。
- 若没有合适单测，先在 `test/` 下新增或扩展一个最小 GTest 用例。
- 若已有测试不完整（如路径覆盖不足、关键指标缺失、复现性差），先补齐再压测。
- 覆盖标准应足以支持优化前后判断：
  - 直接命中目标路径
  - 命令与环境设置可复现
  - 输出可比较的关键指标/计数器

## 4）执行变更前最小路径基准（必选）
- 选择能直接反映目标优化点的最小 benchmark 路径。
- 代码变更前，使用 Google Benchmark 在固定条件下跑基线：
  - 固定 benchmark filter/workload
  - 固定运行环境（CPU 绑核/线程数/构建模式）
  - 尽可能设置重复次数
- 将命令、环境、基线结果记录到 `perf_report/baseline/`。

## 5）定位主导瓶颈
- 优先处理主要贡献项：
  - 热路径拷贝/分配
  - 锁竞争/临界区过大
  - 队列背压失控或无界增长
  - 算法复杂度问题
- 在没有瓶颈证据前，避免大范围重写。

## 6）实施最小高收益改动
- 从能推动目标指标的最小改动开始。
- 除非需求要求改变行为，否则保持正确性与外部可观察行为不变。
- 优先选择易于回滚、改动边界清晰的方案。

## 7）执行变更后基准并对比（必选）
- 使用与步骤 4 完全相同的 Google Benchmark 命令再次执行。
- 基于具体数值和运行配置进行前后对比。
- 尽可能包含波动指标（mean/stddev/CV）。
- 明确给出结论：
  - success：达到或超过阈值
  - failed/not proven：无明确提升，或在噪声范围内，或出现回退
- 没有前后对比证据，不得宣称优化成功。
- 将对比输出保存到 `perf_report/baseline/`。
- 若结果为 `failed/not proven`，必须立即回滚该优化项代码改动。
- 回滚后，在 `todo.md` 将该项标记为 `failed`，并注明日期与简要原因。
- 除非用户后续明确要求继续该失败项，否则默认跳过并执行下一个 TODO 项。

## 8）使用 GTest 做功能/单元验证（必选）
- 对成功或暂时接受的优化项，运行覆盖改动行为的相关 GTest。
- 若行为面有变化且无对应 GTest，先在 `test/` 下补最小 GTest 后再运行。
- 若步骤 7 判定为 `failed/not proven` 且已完成回滚，则该项跳过本步骤并记录跳过原因。
- 报告通过/失败（或跳过）结果，以及未覆盖风险。

## 9）生成优化报告（必选）
- 优化与验证后，在 `perf_report/` 下生成 markdown 报告。
- 报告必须且仅包含以下章节：
  1. 优化的地方
  2. 为什么要这样优化
  3. 优化的效果
  4. 相关的测试用例
- 第 3 节必须包含 benchmark 前后结果和 success/failure 判断。
- 第 4 节必须包含 benchmark 与 gtest 命令，以及引用的 `test/` 下测试文件。

## 10）同步文档与任务状态
- 更新 TODO/任务状态（`success` / `failed` / `skipped`）。
- 对失败项保留 `todo.md` 失败标记；除非用户明确要求，否则不重试。
- 若行为或配置有变化，同步更新 README/配置文档。
- 保证性能报告可复现（命令 + 参数 + 环境）。
- 在每次工作流结束时（无论 `success`/`failed`/`skipped`），必须在对用户的最终回复中列出“剩余待优化项清单”。
- 清单必须包含每项的唯一 ID 与一句话摘要，并按优先级分组（P0/P1/P2）展示。
- 若无剩余项，必须明确写出“无剩余待优化项”。

## 11）提交纪律
- 将相关优化改动作为一个有边界的提交。
- 提交信息遵循仓库风格，意图清晰且简洁。
- 除非用户明确要求，不提交基准产物文件。
