# benchmark-policy

## 1. 目标

为性能优化提供可复现、可比较、可审计的 benchmark 基线与复测规范。

没有可复现 benchmark，就没有可信性能结论。

## 2. 选择顺序

benchmark 或性能测试路径按以下顺序确定：

1. 用户显式指定
2. 仓库现有 benchmark / perf 命令
3. CI 或文档中已有的性能命令
4. 为目标路径补一个最小可复现 benchmark

优先复用已有路径，不要为了“统一”重建工具链。

## 3. 覆盖要求

用于本工作流的 benchmark 必须满足：

- 直接命中目标路径，而不是只测外围流程
- 输入规模与业务场景相关，至少能放大目标瓶颈
- 命令可复现
- 输出包含明确指标
- 可重复运行

若现有 benchmark 不满足以上要求，先补齐或新建最小用例，再做性能修改。

## 4. 最小补齐原则

新增 benchmark 时遵循：

- 只覆盖当前 TODO 对应路径
- 优先单热点、低依赖、低噪声
- 不顺手扩展为完整压测框架
- 命名直接反映目标路径和指标

## 5. 运行规范

baseline 与 after 必须尽量保持一致：

- 相同代码路径
- 相同输入数据
- 相同 workload
- 相同线程数 / 并发度
- 相同构建模式
- 相同关键环境变量
- 相同机器或等价环境

若不能完全一致，必须在报告里明确差异，并降低结论强度。

## 6. 重复运行与波动

优先重复运行，不少于 3 次；若仓库已有更高标准，遵循仓库标准。

每次运行至少记录：

- 主指标
- 样本次数或重复次数
- 最小值 / 最大值 / 平均值
- 可得的波动信息，例如标准差、方差、CV、置信区间

若工具本身不提供统计信息，至少保存多次原始结果并手工汇总。

## 7. 环境记录

每次 benchmark 必须记录：

- 执行命令
- 工作目录
- 构建模式
- CPU / 内存 / 并发相关设置
- 关键环境变量
- 输入数据说明
- 运行时间

若项目对 CPU 绑核、频率控制、NUMA、容器限制敏感，必须一并记录。

## 8. 输出要求

每次 benchmark 至少产出两类内容：

- 原始输出
- 汇总结果

建议放在仓库已有报告目录；若没有约定，按主 skill 的默认候选路径创建报告目录。

## 9. 禁止事项

- 不要在变更后才第一次跑 benchmark
- 不要用不同命令比较前后结果
- 不要混用多个输入规模再直接宣称提升
- 不要只记录“快了/慢了”
- 不要忽略明显噪声和环境漂移

## 10. 通过门槛

只有在以下条件同时满足时，benchmark 数据才可用于下游判断：

- baseline 已存在
- after 已存在
- 两者可比较
- 波动已记录或已解释
- 数据与目标指标一致
